# Hermes 操控博客系统使用手册

适用系统：读写札记博客管理 API。Hermes 通过私有 Docker 网络中的受控 MCP 工具操作博客，权限覆盖文章、日常、媒体、关于页、分类标签、评论、注册用户、书架、微信读书同步、数据库快照和审计记录，但不具备主机 Shell、Docker 或数据库直连权限。

## 1. 推荐的操作模板

向 Hermes 交代博客任务时，尽量包含以下五项：

```text
对象：日常 / 文章 / 书籍 / 评论 / 关于页
动作：创建 / 查询 / 修改 / 发布 / 删除 / 同步
内容：正文、标题、图片、分类等
状态：草稿或直接发布
保护：是否先预览、是否先快照、完成后如何核验
```

最稳妥的默认说法：

```text
先读取当前内容并给我变更预览，不要修改；我确认后再执行。
```

明确要直接执行时：

```text
这是明确的发布请求，可以直接执行。完成后重新读取核验，并返回内容 ID、状态和公开链接（若有）。
```

## 2. 发布“日常”

### 纯文字日常

```text
发布一条日常：
内容：今天重新整理了书桌，也把最近想读的书排了顺序。
地点：山东
状态：直接发布
要求：保留原句，不扩写；完成后返回 ID 和链接。
```

### 带图片日常

先在微信中发送图片，再发送：

```text
把我刚发的 3 张图片用于一条日常，按发送顺序排列。
正文：夜色一落，拎上手电就出发。树下转一圈，满载而归。
地点：山东
直接发布。发布前确认已识别到 3 张图，完成后重新读取核验。
```

### 先存草稿

```text
把下面内容保存为日常草稿，不公开。整理 Markdown 后把预览发给我：……
```

### 日常 Markdown

日常正文支持常用 Markdown，包括标题、粗体、斜体、列表、引用、链接、行内代码和代码块。`#话题` 会成为可点击的话题入口。

推荐写法：

```markdown
**山东人的赶海**

夜色一落，拎上手电就出发。

#赶海 #夏夜
```

图片应作为微信附件交给 Hermes 上传，不要把本地文件路径或 Markdown 图片占位符直接写进正文。

## 3. 查询、修改和删除日常

不知道 ID 时先让 Hermes 列出候选项：

```text
列出最近 5 条日常，给出 ID、发布时间、状态和正文前 40 字，不要修改。
```

修改示例：

```text
读取日常 ID 12，把第一段改为“……”，其他文字和图片保持不变。先给差异预览，等我确认。
```

删除示例分两步：

```text
读取日常 ID 12，告诉我正文摘要、图片数量和发布时间，暂时不要删除。
```

确认无误后再发送：

```text
我确认永久删除日常 ID 12 及其托管图片，直接执行并核验结果。
```

“删除最近那条”不够安全。永久删除时尽量始终写明资源类型和 ID。

## 4. 管理博客文章

### 创建草稿

```text
创建一篇博客文章草稿：
标题：……
摘要：……
分类：……
标签：……、……
正文：使用我下面提供的 Markdown
要求：不要发布；创建后返回 ID、slug 和预览摘要。
```

### 发布文章

```text
读取文章 ID 8，检查标题、摘要、封面、分类、标签和正文是否完整。
先列出检查结果；如果没有阻断问题，再将它发布，并重新读取确认状态。
```

### 修改文章

```text
读取文章 ID 8，只调整二级标题层级和段落空行，不改写内容含义。
先展示修改摘要，等我确认后更新。
```

Hermes 会先读取文章的最新更新时间，再带并发保护提交更新，避免覆盖你刚在后台完成的修改。

### 封面图片

```text
把我刚发的图片上传为文章封面，关联到文章 ID 8。不要修改正文，完成后重新读取核验封面地址。
```

## 5. 管理书架与微信读书

### 手动同步微信读书

```text
现在同步一次微信读书。完成后告诉我新增、更新、归档和失败数量；不要连续重试。
```

服务器的定时任务负责日常同步，微信中的这条指令用于需要立即刷新或排查时手动触发。

### 查询阅读状态

```text
列出当前正在读的书，按最近更新时间排序，给出书名、进度、评分和最近笔记。
```

### 修改展示信息

```text
找到《书名》，先读取完整信息。把它设为置顶，评分改为 4.5，其他同步字段保持不变。
```

可管理的书架信息包括阅读状态、进度、评分、分类、最近感想、公开或私密、置顶和归档。微信读书下一次同步可能覆盖来源于平台的同步字段；手工添加的书摘和感想会保留。

### 添加划线或感想

```text
给《书名》添加一条书摘：
原文：……
我的感想：……
章节：……
先确认匹配到的书名和作者，再添加。
```

```text
给《书名》添加一条读后感：……
```

书名存在多个候选时，要求 Hermes 列出候选，不要猜测内部 source ID。

## 6. 评论、分类与关于页

### 审核评论

```text
列出待审核评论，只显示评论 ID、文章、昵称、时间和内容摘要，不显示邮箱。
```

```text
批准评论 ID 21，完成后重新查询确认它已不在待审核列表。
```

删除评论也属于永久操作，应先读取，再明确说“永久删除评论 ID …”。

### 分类与标签

```text
列出所有分类和标签及其使用次数，不要修改。
```

```text
创建分类“随笔”，如果同名分类已存在则不要重复创建。
```

只有未被文章使用的标签可以删除。删除分类会解除文章关联，因此必须先确认受影响文章数量。

### 关于页

```text
读取当前关于页。把我提供的经历补充到“现在”一节，保持原有 Markdown 结构和语气，先给差异预览，不要直接更新。
```

## 7. 管理注册用户

查询用户：

```text
列出最近注册的 10 位用户，只显示用户 ID、昵称、角色、状态和注册时间，不要显示完整邮箱。
```

修改昵称：

```text
读取用户 ID 8，把显示名改为“新昵称”，其他字段保持不变，完成后重新读取核验。
```

角色和状态属于高影响权限操作，建议分两步：

```text
读取用户 ID 8，告诉我当前角色、状态和最近登录时间，暂时不要修改。
```

确认后：

```text
我确认把用户 ID 8 设为管理员。只修改角色，完成后重新读取核验。
```

```text
我确认停用用户 ID 8。说明影响后执行，完成后重新读取核验。
```

Hermes 不应在微信回复中展示完整邮箱、验证码、会话或令牌。系统会拒绝停用或降级最后一名有效管理员。

服务器端每次创建新用户后，会通过可靠通知队列让 Hermes 向当前绑定的微信会话发送注册提醒。Hermes 暂时不可用时不会阻断注册，通知会自动重试。

## 8. 批量操作、快照与审计

修改多个资源前使用：

```text
先创建一个标签为 before-batch-edit 的数据库快照。确认快照成功后，再把以下 6 篇文章的分类统一调整为“随笔”。逐项核验，任何一项失败就停止并汇报。
```

查看最近操作：

```text
列出最近 20 条 Hermes 博客管理审计记录，按时间倒序总结成功、失败和操作对象，不要暴露敏感信息。
```

快照用于内容数据库恢复点，不等同于整台服务器备份。恢复操作不在微信博客工具的常规能力中，需要受控运维流程处理。

## 9. 建议直接收藏的指令

```text
检查博客管理接口状态，告诉我版本、内容数量和安全保护是否正常，不做修改。
```

```text
列出最近 5 条已发布日常和 5 篇已发布文章，只返回标题或摘要、ID、时间和链接。
```

```text
检查待审核评论并给出审核建议，不要自动批准或删除。
```

```text
同步微信读书一次，返回精确统计；失败时保留原数据，不要高频重试。
```

```text
先创建数据库快照，再执行我接下来确认的批量修改。
```

```text
读取目标后再修改；只改我点名的字段，其余保持不变；完成后重新读取核验。
```

## 10. Hermes 的默认安全行为

- 意图明确写“发布”时，可以直接发布；意图含糊时应先保存草稿或询问。
- 更新前先读取目标，并使用最新更新时间做并发保护。
- 删除必须来自明确删除请求，并经过专用确认机制。
- 批量或高影响修改前先创建数据库快照。
- 写操作完成后重新读取目标，确认状态、正文、图片和更新时间。
- 不在微信回复中暴露管理令牌、内部地址、评论者邮箱或服务器路径。
- 修改用户角色或状态前必须获得明确确认，并保护最后一名有效管理员。

## 11. 一句话原则

先说清对象和范围，再说“草稿、预览或直接发布”；修改前读取，批量前快照，删除时写明 ID，完成后要求重新核验。
